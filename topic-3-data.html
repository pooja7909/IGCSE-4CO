⚡ 3.1.1 Why do computers use binary?

Computers store and process every piece of information — numbers, text, images, sound, even the program instructions themselves — as nothing more than streams of 1s and 0s. To understand why, we need to look inside the processor.

A modern CPU contains billions of transistors. A transistor is a tiny electronic switch with only two states:

🔆 ON (1)
Current is flowing through the transistor — voltage is high (typically around 5V or 3.3V).
⚫ OFF (0)
No current flowing — voltage is low (close to 0V).

This system with two clearly distinct states is called digital. Compare it with an analogue system like a dimmer switch — a continuous range of brightness from off to fully on. Analogue signals would be much harder for a computer to use because:

  • Tiny variations in voltage caused by noise/heat could be misread as different values
  • Sending and receiving systems would need to be extremely precise
  • Communication errors would be frequent

With binary, there are only two states to distinguish — even a degraded signal is usually clearly either "high" or "low", so communication is reliable and there are no misunderstandings.

🎓 Teacher note A common exam question asks: "Why is binary used in computer systems?". Answers must link two states of transistors to two binary digits (0 and 1). Saying "because computers are electronic" is not enough for full marks.
🧮 3.1.2 Place values — binary vs denary

Both systems are positional: a digit's value depends on its position. The difference is the base:

Denary (base 10)
Each column is 10× the column to its right.
Powers of 10: 1, 10, 100, 1000…

Example: 3639 = (3×1000) + (6×100) + (3×10) + (9×1)
Binary (base 2)
Each column is 2× the column to its right.
Powers of 2: 1, 2, 4, 8, 16…

Example: 1011 = (1×8) + (0×4) + (1×2) + (1×1) = 11

For an 8-bit binary number (one byte), the place values you must memorise are:

2⁷2⁶2⁵2⁴2³2²2¹2⁰
1286432168421
🧠 Memory aid Start from 1 on the right and double each time: 1 → 2 → 4 → 8 → 16 → 32 → 64 → 128. Practise writing these out from memory until you can do it in < 5 seconds — examiners will not give you the table!

The largest 8-bit unsigned number is therefore: 128+64+32+16+8+4+2+1 = 255. Combined with zero, an 8-bit byte represents the 256 different values from 0 to 255.

🧠 The "2ⁿ rule" With n bits you can represent 2ⁿ different values. This appears everywhere in this topic:
• 1 bit → 2¹ = 2 patterns (just 0, 1)
• 4 bits (nibble) → 2⁴ = 16 patterns
• 7 bits (ASCII) → 2⁷ = 128 patterns
• 8 bits (byte) → 2⁸ = 256 patterns
• 16 bits → 2¹⁶ = 65,536 patterns
• 24 bits ("true colour") → 2²⁴ ≈ 16.7 million patterns
➡️ 3.1.3a Converting binary → denary

Method: write the place-value table, write each bit underneath, then add the place values that have a 1 above them.

📐 Worked example — convert 10101101 to denary
1286432168421
10101101
Add the columns with a 1: 128 + 32 + 8 + 4 + 1 = 173
⬅️ 3.1.3b Converting denary → binary

Method (left-to-right subtraction): for each column, ask "Is the remaining number greater than or equal to this place value?". If YES, write 1 and subtract the place value. If NO, write 0 and move on.

📐 Worked example — convert denary 213 to binary
1213 ≥ 128? Yes → write 1. Remainder = 213 − 128 = 85
285 ≥ 64? Yes → write 1. Remainder = 85 − 64 = 21
321 ≥ 32? No → write 0.
421 ≥ 16? Yes → write 1. Remainder = 21 − 16 = 5
55 ≥ 8? No → write 0.
65 ≥ 4? Yes → write 1. Remainder = 5 − 4 = 1
71 ≥ 2? No → write 0.
81 ≥ 1? Yes → write 1. Remainder = 0. ✓
Answer: 213 = 11010101
⚠️ Always check your answer. Add up the place values where you wrote a 1: 128+64+16+4+1 = 213 ✓. This is also called the "place-value subtraction method".
➕ 3.1.4a Binary addition

Works exactly like denary addition, but the carry-over rule kicks in much sooner because the largest binary digit is 1 (in denary it's 9). The four cases you need to know:

  carry:   1  1  1  1     1

The carry "fell off the left end" — the answer needs 9 bits but we only have 8. This is an overflow error.
⚠️ 3.1.4b Overflow errors

Overflow happens when the result of a calculation requires more bits than the system can store. For 8-bit addition, overflow occurs if the answer needs 9 bits (i.e. the carry "falls off" the leftmost column).

When overflow happens, the CPU sets a special overflow flag in its status register so the processor knows an error has occurred. If the program doesn't check this flag, the wrong (truncated) answer is used — sometimes with catastrophic consequences.

🚀 Real disaster — Ariane 5 (1996) On 4 June 1996, the European Space Agency's Ariane 5 rocket exploded just 40 seconds after launch. The cause? A 64-bit floating-point number representing the rocket's horizontal velocity was converted to a 16-bit signed integer. The value was too large to fit — overflow occurred — and the guidance computer interpreted the garbage result as an extreme course deviation. It tried to correct, the rocket ripped itself apart aerodynamically, and the self-destruct triggered. Estimated cost: $370 million. The Patriot missile defence failure (1991, killing 28 soldiers) was a similar overflow story.
➖ 3.1.2a Signed integers — Sign and Magnitude

So far we've only handled positive numbers. But how do we represent negative numbers in binary? We can't just add a minus sign — there is no "minus sign" in binary, only 0 and 1.

The first (and simplest) approach is sign and magnitude. We sacrifice the MSB (leftmost bit) to act as a "sign bit":

  • MSB = 0 → the number is positive
  • MSB = 1 → the number is negative
  • The remaining 7 bits hold the magnitude (the size)
00011100 → MSB is 0, so positive. Magnitude = 28 → represents +28
10011100 → MSB is 1, so negative. Magnitude = 28 → represents −28

The largest positive 8-bit number using sign and magnitude is now only +127 (because one bit is used for the sign), and the most negative is −127.

⚠️ Two big problems with sign and magnitude:
  1. Two zeros: 00000000 = +0 and 10000000 = −0. The same number has two representations — nonsensical for the CPU.
  2. Arithmetic doesn't work: Adding the binary for +28 to the binary for −28 should give 0, but using sign and magnitude it gives a result equivalent to −56. The CPU would need special hardware just to handle signs.
🔄 3.1.2b Two's Complement (the modern method)

Two's complement is the method used by virtually every modern computer. It solves both problems above and lets us do subtraction by addition.

How to convert a positive number to its negative in two's complement:

  1. Flip every bit (0 → 1, 1 → 0). This is called the one's complement.
  2. Add 1 to the result.
📐 Worked example — find the 8-bit two's complement of −28 Step 1: write +28 in binary → 00011100
Step 2: flip every bit → 11100011
Step 3: add 1 → 11100100
Therefore −28 = 11100100 in 8-bit two's complement.

To convert back from two's complement to denary, you can use the place-value method, but the MSB now has a negative value:

−1286432168421
11100100

Sum: −128 + 64 + 32 + 4 = −28 ✓

The range of 8-bit two's complement is therefore −128 to +127 (still 256 values, no duplicated zero, and arithmetic works correctly).

🧠 Quick check The MSB still tells you the sign instantly:
• MSB = 0 → positive (use normal place values, the MSB has value 0×128 = 0)
• MSB = 1 → negative (the MSB contributes −128)
📊 3.1.2c Highest and lowest values — for ANY number of bits

One of the most common exam questions asks: "What is the largest/smallest value that can be represented using n bits in (unsigned / sign-and-magnitude / two's complement)?". You must be able to answer this for any value of n — not just 8 bits. The formulae below work universally.

THE THREE KEY FORMULAE For an n-bit binary number:

Unsigned integers
• Smallest value: 0
• Largest value: 2ⁿ − 1
• Total distinct values: 2ⁿ

Sign and magnitude (1 bit for sign, n−1 bits for magnitude)
• Most negative value: −(2ⁿ⁻¹ − 1)
• Most positive value: +(2ⁿ⁻¹ − 1)
• Total distinct values: 2ⁿ − 1 (because +0 and −0 are duplicates!)

Two's complement
• Most negative value: −(2ⁿ⁻¹)
• Most positive value: +(2ⁿ⁻¹ − 1)
• Total distinct values: 2ⁿ (no duplicate zero)

🧠 The "one extra negative" rule Two's complement always has one more negative value than positive value. Why? Because there's only one zero (00000000) — it sits on the "positive" side, leaving one extra slot on the negative side.

• 8-bit two's complement: −128 to +127 → 128 negatives, 127 positives, 1 zero.
• 4-bit two's complement: −8 to +7 → 8 negatives, 7 positives, 1 zero.
• Always: the most negative number has no positive equivalent. Try to find the two's complement of −128 in 8 bits — you get back −128! This is sometimes called the "two's complement gotcha".
🧠 Why sign & magnitude loses one value Sign and magnitude has +0 and −0 — the same number stored two different ways. That's one wasted bit pattern. So sign and magnitude can only represent 2ⁿ − 1 distinct values (e.g. 255 with 8 bits, not 256).
📐 Worked example — 6-bit ranges With n = 6:
• 2ⁿ = 2⁶ = 64
• 2ⁿ⁻¹ = 2⁵ = 32
• 2ⁿ⁻¹ − 1 = 31

Therefore:
• Unsigned: 0 to 63 (= 2⁶ − 1)
• Sign & magnitude: −31 to +31
• Two's complement: −32 to +31
📐 Worked example — finding bit-width from a range "What is the minimum number of bits needed to represent the value −500 in two's complement?"

Two's complement n bits can store down to −2ⁿ⁻¹. We need −2ⁿ⁻¹ ≤ −500, i.e. 2ⁿ⁻¹ ≥ 500.
• 2⁸ = 256 → too small
• 2⁹ = 512 → big enough ✓
So n − 1 = 9, therefore n = 10 bits.

Quick check: 10-bit two's complement range is −512 to +511. ✓ Yes, −500 fits.

⬅️➡️ 3.1.4c Binary shifts

A binary shift moves all the bits of a number left or right by a specified number of places. There are two flavours — you must understand both.

In denary, shifting digits left by 1 place multiplies by 10 (e.g. 13 → 130). Shifting right by 1 divides by 10 (130 → 13). The same idea works in binary, but each shift multiplies/divides by 2 instead of 10.

⬅️ Logical shift (unsigned)
Used for unsigned (positive only) integers.

Left shift by n: all bits move n places left, n zeros fill in on the right, leftmost bits fall off. Effect: multiply by 2ⁿ.

Right shift by n: all bits move n places right, n zeros fill in on the left, rightmost bits fall off. Effect: divide by 2ⁿ (rounded down).
📐 Arithmetic shift (signed)
Used for signed two's complement integers.

Left shift: the MSB stays in place (it must stay there to keep the sign correct); the other bits shift left and a 0 fills in on the right.

Right shift: bits shift right, but copies of the MSB fill in on the left. This preserves the sign — a negative number stays negative when divided by 2.
📐 Logical shift example — 00010100 (= 20) shifted left by 2 
Before:  0 0 0 1 0 1 0 0    (= 20)
Shift left by 2:
After:   0 1 0 1 0 0 0 0    (= 80)
Check: 20 × 2² = 20 × 4 = 80 ✓
📐 Arithmetic right shift example — 11011100 (= −36) shifted right by 1 
Before:  1 1 0 1 1 1 0 0    (= −36 in two's complement)
Right shift 1, MSB copied on the left:
After:   1 1 1 0 1 1 1 0    (= −18 in two's complement)
Check: −36 ÷ 2 = −18 ✓. The MSB (1) was copied across so the number stays negative.
⚠️ Shifts can lose precision! When you right-shift 00100001 (= 33) by 1 place, you get 00010000 (= 16). But 33 ÷ 2 = 16.5! Binary integers can't hold fractions, so the 0.5 is lost.
#️⃣ 3.1.5 Hexadecimal

Hexadecimal is a base-16 number system. Place values are powers of 16: 1, 16, 256, 4096… It uses 16 digits — the familiar 0–9 plus the letters A, B, C, D, E, F for the values 10–15.

Denary0123456789101112131415
Hex0123456789ABCDEF
❓ Why use hexadecimal at all?

Computers never actually use hex. It's purely for human convenience. As computers moved from 8-bit to 32-bit and 64-bit, binary numbers got far too long for humans to read, copy or debug:

  • A 32-bit memory address as binary: 11111111000000000000000000001100 😵
  • The same address as hex: FF00 000C 😊

One hex digit always represents exactly 4 binary digits (a nibble), because 2⁴ = 16. Two hex digits = 8 bits = 1 byte. This makes binary↔hex conversion mechanical and error-free.

🧠 Three reasons hex is used • Shorter — 4× more compact than binary.
• Easier to read aloud, copy and remember.
• One-to-one correspondence — each hex digit cleanly maps to 4 bits, so converting is just splitting the binary into nibbles.

Real uses of hex you'll see:

  • Web colours / CSS: #FF0000 (red), #00FF00 (green), #0000FF (blue), #F0F8FF (Alice blue). Each pair encodes 8 bits per primary colour.
  • MAC addresses: the unique ID of a network card, e.g. 3C:5A:B4:E2:F1:09.
  • Memory addresses in debuggers and assembly code.
  • Unicode code points: the heart emoji ❤ is U+2764.
  • Error codes: Windows "blue screen of death" codes like 0x0000007B.
↔️ Converting binary ↔ hex

Binary → Hex:

  1. Split the binary into nibbles (4-bit groups) from the right.
  2. Convert each nibble to its denary value (0–15).
  3. Write each value as a hex digit (10 = A, 11 = B, …, 15 = F).
📐 Worked example — 11100110 to hex Split into nibbles: 1110 | 0110
Convert each nibble: 1110 = 8+4+2 = 14,   0110 = 4+2 = 6
Write as hex digits: 14 = E,   6 = 6
Answer: E6

Hex → Binary: reverse the process — convert each hex digit to a 4-bit binary number, then join them up.

📐 Worked example — AC to binary Convert each digit: A = 10 = 1010,   C = 12 = 1100
Join: 10101100

🔬 Deep Dive — Binary at the hardware level

What is a transistor, really? A modern transistor (specifically a MOSFET) is microscopic — just 5 nanometres wide on the latest chips. That's 10,000× thinner than a human hair. It works like a faucet: a tiny voltage on its "gate" pin decides whether electricity can flow between its "source" and "drain" pins. When the gate voltage is high, current flows (state = 1). When low, current is blocked (state = 0). The newest Apple M3 chip contains over 25 billion transistors in a piece of silicon smaller than a postage stamp.

How does the CPU "know" what a bit pattern means? It doesn't — context decides. The byte 01000001 could be:

  • The number 65 (as an unsigned integer)
  • The character 'A' (as ASCII)
  • A pixel that's a dim red (as a colour byte)
  • A program instruction telling the CPU to do something
  • Part of a larger number (e.g. one byte of a 32-bit integer)

The program that's running tells the CPU which interpretation to use. The same pattern in different memory locations means completely different things — this is one of computer science's most important ideas.

💡 The "binary milometer" analogy (for two's complement) Imagine an old car milometer with three digits, going from 000 to 999. What does the milometer read if you reverse the car by 1 mile starting from 000? It "rolls back" to 999. So 999 is acting like "−1". 998 = "−2". 997 = "−3". This is exactly how two's complement works! In 8-bit two's complement, the pattern 11111111 represents −1, 11111110 represents −2, and so on. The "milometer" wraps around — and remarkably, addition just works.
❌ Common student misconception "An 8-bit byte can store numbers from 0 to 256." WRONG. It stores 256 different values, but those are 0–255 (because 0 is one of them). Always remember: n bits → 2ⁿ patterns → values 0 to 2ⁿ−1.
🌍 Where you've seen hexadecimal in real life • CSS / web design: background-color: #FF5733;
• MAC addresses: every wi-fi adapter has one, e.g. 3C:5A:B4:E2:F1:09
• Memory dumps: when a program crashes, debuggers show memory contents as hex
• HTML character entities: &#x1F600; = 😀
• Bitcoin / Ethereum addresses: long hex strings like 0x742d35Cc66...
• Git commit hashes: 7-character hex IDs like a1b2c3d
⚙️ Pseudocode — Convert denary to 8-bit binary
Algorithm 1
# Convert a denary number (0-255) to 8-bit binary string RECEIVE number FROM (INTEGER) KEYBOARD SET binary TO "" SET placeValues TO [128, 64, 32, 16, 8, 4, 2, 1] FOR i FROM 0 TO 7 DO IF number >= placeValues[i] THEN SET binary TO binary + "1" SET number TO number - placeValues[i] ELSE SET binary TO binary + "0" END IF END FOR SEND binary TO DISPLAY
⚙️ Pseudocode — Convert 8-bit binary to denary
Algorithm 2
# Convert an 8-bit binary string to denary RECEIVE binary FROM (STRING) KEYBOARD SET denary TO 0 SET placeValues TO [128, 64, 32, 16, 8, 4, 2, 1] FOR i FROM 0 TO 7 DO IF binary[i] = "1" THEN SET denary TO denary + placeValues[i] END IF END FOR SEND denary TO DISPLAY
⚙️ Pseudocode — Find the two's complement
Algorithm 3
# Convert positive binary to its negative form (two's complement) RECEIVE binary FROM (STRING) KEYBOARD SET flipped TO "" # Step 1: flip every bit FOR i FROM 0 TO LENGTH(binary) - 1 DO IF binary[i] = "0" THEN SET flipped TO flipped + "1" ELSE SET flipped TO flipped + "0" END IF END FOR # Step 2: add 1 (using binary addition with carry) SET result TO "" SET carry TO 1 FOR i FROM LENGTH(flipped) - 1 DOWN TO 0 DO SET sum TO INT(flipped[i]) + carry IF sum = 2 THEN SET result TO "0" + result SET carry TO 1 ELSE SET result TO STR(sum) + result SET carry TO 0 END IF END FOR SEND result TO DISPLAY
💼 Industry insight — Why this still matters Two's complement isn't a curiosity — it's how every modern CPU on Earth represents negative numbers. The exact same Intel x86 instruction (ADD) handles both signed and unsigned addition, because two's complement is designed so that the underlying binary math is identical. The CPU just sets different "flags" depending on which interpretation the program wants. This elegance is why two's complement has won out over the alternatives for 60+ years.

🔤 3.2.1a Representing text — ASCII

When you press a key on the keyboard, hardware sends a binary code to the processor. The computer matches this code against its character set to decide which character was typed. For two computers to understand each other, they must share the same character set — otherwise typing "Hello" on one machine could come up as gibberish on another.

The earliest widely-adopted standard was ASCII (American Standard Code for Information Interchange), introduced in 1963. ASCII originally used 7 bits, giving 2⁷ = 128 different characters. That's enough for:

  • 26 upper-case letters A–Z (codes 65–90)
  • 26 lower-case letters a–z (codes 97–122)
  • 10 digits 0–9 (codes 48–57)
  • Punctuation marks: . , ; : ? ! ' " ( ) etc.
  • Control characters (codes 0–31 and 127) — non-printing instructions like carriage return, line feed, backspace, tab, escape, delete.
🧠 The "magic 32" between cases Lower-case codes are exactly 32 more than the matching upper-case codes:
• 'A' = 65, 'a' = 97 (65 + 32)
• 'Z' = 90, 'z' = 122 (90 + 32)
This is no coincidence — bit 5 (the 6th bit, value 32) flips the case! This makes case conversion in programs trivially fast.

Important consequence: the digit '5' (the character) is not the same thing as the number 5! '5' has ASCII code 53, stored as binary 00110101. The number 5 is stored as 00000101. Programs must convert between them when reading user input.

ASCII is also designed so that characters are grouped logically. Sorting strings alphabetically just means sorting by their ASCII codes — which is why most programming languages give the right answer when you compare strings.

🎓 Why this matters for the Caesar cipher When you write a Caesar cipher program, you don't move letters around in some special way — you just add the shift value to the ASCII code. For example, 'D' (68) + 3 = 71 = 'G'. The hint section in past Edexcel papers explicitly tells you this.
🌍 3.2.1b Extended ASCII and Unicode

128 characters is not enough for the world. As computing spread internationally in the 1980s, people needed letters with accents (é, ñ, ü), Greek letters, mathematical symbols, line-drawing characters for early text-based menus, currency symbols (£, €, ¥) and so on.

Extended ASCII added an 8th bit, giving 2⁸ = 256 characters. The first 128 codes were still standard ASCII, leaving 128 extra slots for the new characters. However, there was no standardisation:

  • IBM filled the extra slots with one set of characters
  • Apple chose a different set
  • Atari, Commodore, and various national variants each had their own

So a document with "café" written on an IBM PC would arrive on a Mac looking like "caf©" or worse. This was a serious compatibility nightmare.

The solution was Unicode (developed by the Unicode Consortium from 1991 onwards). Unicode:

  • Uses 2 or 4 bytes per character — enough for over a million different code points
  • Covers every known writing system: Latin, Cyrillic, Greek, Arabic, Hebrew, Chinese, Japanese, Korean, Hindi, Tamil, Cherokee, Egyptian hieroglyphs, even fictional scripts like Tolkien's Tengwar in unofficial extensions
  • Includes emoji 😀, mathematical symbols ∑ ∫ √, currency symbols, dingbats and special control characters
  • Is backwards compatible with ASCII: Unicode codes 0–127 are identical to ASCII codes 0–127
🧠 ASCII vs Unicode — the quick comparison • ASCII: 7 (or 8) bits per character → 128 (or 256) characters → English-only
• Unicode: 16 or 32 bits per character → over 1 million possible characters → every language plus emoji
Trade-off: Unicode files are 2–4× bigger than pure ASCII, but you can write in any language.
🖼️ 3.2.2a Bitmap images — pixels

A bitmap image is built from a rectangular grid of tiny squares called pixels (picture elements). Each pixel stores the colour at one point of the image as a binary number. When you zoom in far enough on any digital photograph, you'll see the individual coloured squares.

Three properties define a bitmap image:

📐 Image size
Written as width × height pixels, e.g. 640 × 480. A 640×480 image contains 307,200 pixels.
🔍 Resolution
Pixels per inch (PPI) when displayed. More PPI in the same area = smaller pixels = more detail visible.
🎨 Colour depth
The number of bits used to store each pixel's colour. Determines how many different colours are possible.
⚠️ Don't confuse "image size" and "resolution". Image size is fixed (how many pixels the image contains). Resolution depends on how big you display it. Stretch a 100×66 image to fill your monitor and the resolution drops — same pixels, larger area, less detail visible.

🔊 3.2.3a What is sound? Analogue vs digital

Sound is created by vibrations. Vocal cords, guitar strings or loudspeakers push air molecules back and forth, sending pressure waves through the air. When these waves reach your ear, sensitive hairs in the inner ear vibrate and the brain interprets the signals as sound.

A sound wave is naturally analogue: the air pressure varies continuously over time, taking any value within a range. Two important features of a wave are:

  • Amplitude — how big the wave is (higher amplitude = louder sound)
  • Frequency — how many wave cycles per second, measured in Hz (higher frequency = higher pitch)

Old technologies like vinyl records and magnetic tape store sound as a continuous analogue signal — the depth of the groove or the strength of magnetisation varies exactly in step with the original wave. This is fairly accurate (because there are no "steps" — every nuance of the wave is preserved) but has problems: records scratch, tapes stretch, and copies are always slightly worse than the original.

Computers can't store continuous values — only discrete 1s and 0s. So digital sound recording takes regular "snapshots" of the wave's amplitude. Played back fast enough, these snapshots produce a sound that seems continuous to the ear.

🎧 3.2.3b Sampling

Sampling = taking measurements of the sound wave's amplitude at regular, very short intervals of time. Each measurement is converted to a binary number by an Analogue-to-Digital Converter (ADC).

To play the sound back, the binary stream is fed through a Digital-to-Analogue Converter (DAC), which reconstructs an approximation of the original wave and drives a speaker.

Two factors affect the quality of the recording:

⏱️ Sample rate (frequency)
Samples per second, measured in Hz.

CD audio: 44,100 Hz (44.1 kHz)
Blu-ray audio: 96,000 Hz (96 kHz)
Telephone quality: ~8,000 Hz (poor!)

Higher rate = wave reconstructed more accurately = higher fidelity, but bigger file.
🎚️ Bit depth
Bits per sample — how finely we can measure each amplitude.

8 bits → 2⁸ = 256 amplitude levels
16 bits → 2¹⁶ = 65,536 levels (CD standard)
24 bits → 2²⁴ = 16,777,216 levels

Higher bit depth = more accurate amplitude = quieter "dynamic range" is captured.
🎓 Why 44.1 kHz for CDs? The Nyquist–Shannon theorem says you need at least twice the sampling rate of the highest frequency you want to capture. Human hearing tops out around 20 kHz, so a sampling rate slightly above 40 kHz is enough — 44.1 kHz gives a small safety margin and was historically convenient for adapting video equipment to early digital audio mastering.
📉 3.2.4 Limitations of binary representation

No matter how good our digital recording is, it's never exactly the same as the original analogue wave. We're always approximating a smooth curve with a series of stepped values:

  • If the sample rate is too low, fast wave changes between samples are missed. High-frequency content (sharp consonants, cymbal crashes) sounds muddy.
  • If the bit depth is too low, subtle volume changes get rounded — quiet passages sound "stair-step" instead of smooth. This is called quantisation error.

The recording's closeness to the original is called fidelity. "High fidelity" (hi-fi) means the recording sounds almost identical to the live performance.

✅ Advantages of digital audio
• Recording equipment is cheap; home recording is easy
• Files don't deteriorate with playback (unlike vinyl/tape)
• Easily edited with software
• Perfect copies can be made instantly
• Easily transmitted (streamed/emailed/downloaded)
❌ Limitations
• Continuous wave can never be captured perfectly
• Low sample rate or bit depth audibly degrades quality
• Higher quality = bigger files (storage/bandwidth cost)
• Audio enthusiasts argue analogue captures "warmth" digital can't

🔬 Deep Dive — How encoding actually works

What happens between key-press and screen? Let's trace pressing the letter 'A':

  1. Your finger pushes a key. A mechanical switch closes a circuit.
  2. The keyboard's microcontroller detects the switch and looks up its "scan code" in a table.
  3. The scan code is sent via USB as a sequence of binary 1s and 0s to the computer.
  4. The operating system's keyboard driver translates the scan code into a character code — usually Unicode in modern systems (which happens to match ASCII for English letters: 'A' = 65).
  5. The application (e.g. word processor) receives the code 65.
  6. To display it, the program looks up character 65 in the current font — which is itself a file containing pixel patterns or vector outlines.
  7. The font's instructions for 'A' are rasterised into pixels and sent to the GPU.
  8. The GPU writes those pixel colours to the video memory buffer.
  9. The monitor reads the buffer and lights up the corresponding pixels on screen.

Every single step is binary. The whole journey from finger to display takes about 5–10 milliseconds.

💡 The "sampling = flipbook" analogy A flipbook animation works by drawing many static pictures, each slightly different from the last. Flick the pages fast enough and your brain "fills in" the motion. Digital audio is exactly the same: each sample is a snapshot of the wave amplitude, and at 44,100 snapshots per second your ear "fills in" what sounds like a continuous wave. Flick a flipbook too slowly and the motion looks jerky; sample audio too slowly and the recording sounds muddy.
💡 The "graph paper" analogy for bit depth Imagine measuring the height of a wave with a ruler. An 8-bit ruler has 256 tiny marks; a 16-bit ruler has 65,536 marks across the same distance. Both rulers measure the same wave, but the 16-bit ruler can pin down each measurement to a much finer position. The wave you reconstruct from 16-bit measurements is therefore much smoother than the wave from 8-bit measurements.

Why are pixels so common? Bitmap images are conceptually simple (a grid of coloured dots), but they have a fundamental problem: they don't scale. Make a 200×200 photo three times bigger and it looks blurry, because there are no extra pixels to fill the gap — the original ones get stretched. This is why logos and icons are increasingly stored as vector graphics (SVG files), which use mathematical formulas (lines, curves) rather than pixels. Vector images stay crisp at any size — but they're poor for photographs because real-world scenes don't reduce to simple curves.

🌍 What's actually inside common file formats? • .txt file: just a stream of ASCII (or Unicode) bytes. Open in any text editor.
• .bmp: an uncompressed bitmap. A small header followed by raw pixel data (width × height × colour depth ÷ 8 bytes).
• .png: a lossless compressed bitmap. Uses a clever combination of filtering + DEFLATE compression.
• .wav: an uncompressed audio file. A short header followed by raw samples.
• .jpg: a lossy bitmap. Uses the Discrete Cosine Transform to throw away imperceptible detail.
• .mp3: a lossy audio file. Uses psychoacoustic modelling to remove inaudible frequencies.
❌ Common student misconception "A higher-resolution image always has more pixels." Subtle but wrong. Resolution is pixels per inch, not the total number of pixels. A 100×100 image at 300 PPI prints sharp but tiny (1/3 inch square). A 100×100 image at 72 PPI prints fuzzy and larger (1.4 inch square). Image size = how many pixels exist. Resolution = how densely they're packed when displayed.

Why 24 bits (not 16 or 32) for "true colour"? Three reasons:

  • Biological: the human eye can distinguish around 10 million different colours. 24 bits gives 16.7 million — comfortably beyond perception.
  • Symmetrical: dividing 24 bits evenly gives 8 bits per primary colour (R, G, B) — and 8 bits is exactly one byte, which is convenient.
  • Hex-friendly: 24 bits = 6 hex digits, the familiar #RRGGBB format used in CSS and design tools.

🤔 3.3.2a Why compress data?

The World Economic Forum estimates that by 2025, humanity produces 463 exabytes of data every day (≈ 1 exabyte = 1 billion gigabytes). Without compression we'd be drowning in storage costs and network congestion. Specific benefits:

💾 Storage
Compressed files take less space. A phone with 64 GiB can hold ~16,000 MP3 songs but only ~1,600 uncompressed WAV files.
🌐 Bandwidth
Smaller files transfer faster and use less bandwidth. Vital for streaming services serving millions of users simultaneously.
⏱️ Speed
Less time spent uploading/downloading. Web pages load faster. Backups complete sooner.
🚦 Less congestion
Less internet traffic globally → less network congestion → better experience for everyone.

The catch is that the computer must spend a small amount of CPU time to compress and decompress files. For most modern devices this is negligible compared with the time saved.

🟢 3.3.2b / 3.3.3 Lossless compression and RLE

Lossless compression preserves every single bit of the original file — when decompressed, the result is byte-for-byte identical to what was compressed. Used wherever data loss would be catastrophic:

  • Text files (.txt, .doc) — missing characters change meaning
  • Source code (.py, .java, .c) — a missing character can break the program
  • Executable programs (.exe) — corrupt one byte and the app crashes
  • Spreadsheets and databases — bank-balance accuracy matters
  • Lossless image formats: PNG, GIF (good for logos, diagrams, screenshots)
  • General-purpose archivers: ZIP, RAR, 7z, gzip

Lossless compression works by exploiting redundancy — patterns that occur many times in the data. The compressor records the pattern once and refers back to it. In English text, the word "the" appears thousands of times in any book — store it once and refer to it.

The Edexcel spec only requires one specific lossless algorithm: Run-Length Encoding (RLE).

🔄 RLE in detail

RLE looks for runs — consecutive sequences of the same value — and replaces each run with two pieces of information:

  1. The count (how many times the value repeats)
  2. The value being repeated
📐 Worked example — compressing a text string Original (22 characters / 22 bytes if 1 byte per char):
cccmmmmmssssdddccccccc

Identify the runs: 3 c's, 5 m's, 4 s's, 3 d's, 7 c's.
RLE output: 3c5m4s3d7c (10 characters / 10 bytes)

Saving: 22 → 10 bytes ≈ 55% reduction.

RLE is also widely used for simple bitmap images (logos, icons, diagrams). A black-and-white pixel grid has long runs of 'w' (white) and 'b' (black):

📐 RLE for an 8-pixel-wide row of a bitmap letter Original row (8 bytes): w w b b b w w w
Runs: 2 w's, 3 b's, 3 w's
RLE: 2w 3b 3w (6 bytes — saved 25%)

For an 8 × 8 black-and-white bitmap of the number "5", RLE compresses 64 bytes down to about 48 bytes — and the saving grows for larger, blockier images.

⚠️ RLE can backfire. If the data has no repeating runs (e.g. a noisy photograph where every pixel is slightly different), RLE actually makes the file bigger, because it inserts a "1" before every single value. Run-length encoding only works well on highly repetitive data.
🔴 3.3.2c Lossy compression

Lossy compression achieves much bigger size reductions by permanently deleting data that humans usually can't perceive. Once compressed, the original cannot be recovered.

⚠️ NEVER use lossy compression for: text documents, program code, executables, spreadsheets, databases, or any file where every byte matters. One lost character could change "transfer $100" into "transfer $1000" — catastrophic.

Lossy is ideal for media files (photos, audio, video) where small imperfections are imperceptible:

🖼️ JPEG (.jpg) — images
Developed by the Joint Photographic Experts Group.

The algorithm analyses the image and finds areas where many neighbouring pixels have very similar colours. It rewrites those pixels with the same colour value, then exploits the resulting redundancy.

Our eyes can't distinguish subtle colour differences in photos, so it looks identical to the original. Typical reduction: 6 MB → 350 KB (≈ 17× smaller).
🎵 MP3 (.mp3) — audio
MPEG-1 Audio Layer 3.

Uses psychoacoustic modelling: a study of how the human ear works. The algorithm removes:
• Frequencies humans can't hear (below 20 Hz and above 20 kHz)
• Quieter sounds masked by louder simultaneous sounds
• Tiny volume changes the ear can't detect

Typical reduction: 30 MB WAV → 3 MB MP3 (≈ 10× smaller). Modern alternative: AAC (used by Apple).

For video, codecs like H.264, H.265 and AV1 combine JPEG-style image compression with techniques that store only what changes between frames — turning multi-gigabyte source footage into something small enough to stream on a phone.

🔬 Deep Dive — Compression in the real world

How much data does humanity actually produce? Every minute, internet users:

  • Send around 250 million emails
  • Stream over 1 million hours of YouTube video
  • Send hundreds of millions of WhatsApp messages
  • Upload nearly 1 million photos across all platforms

If none of this was compressed, the global internet backbone would have collapsed years ago. Compression is literally what makes the modern internet possible.

💡 The "suitcase packing" analogy Imagine packing for a holiday. Lossless compression is like rolling your clothes instead of folding — same items fit in less space, and when you unpack, you get every single garment back. Lossy compression is leaving some items behind (your fifth pair of socks; the spare phone charger) — your suitcase is much lighter, but those items are gone forever. You'd never leave your passport behind (= text files), but skipping a few socks (= subtle audio details) is fine.
💡 The "shorthand" analogy for RLE A court stenographer writes "the defendant" as "d/" and "for the record" as "ftr". Same meaning, fewer keystrokes. RLE works the same way — instead of writing aaaaaa six times, write "6a" once. The "key" of the shorthand must be agreed in advance, which is why RLE encoders and decoders both follow the same rule (count first, then symbol).

How JPEG actually works (simplified):

  1. The image is split into 8×8 pixel blocks.
  2. Each block is converted from RGB colour to YCbCr (Y = brightness; Cb, Cr = colour differences). The human eye is far more sensitive to brightness than colour, so the colour channels can be downsampled.
  3. A mathematical transform (the Discrete Cosine Transform) converts each block into a list of frequency components — telling us how much "smooth area" vs "fine detail" the block contains.
  4. High-frequency components (fine details our eyes barely notice) are rounded down or removed. This is the lossy step.
  5. The remaining data is then compressed losslessly using Huffman coding.

The amount of detail thrown away is controlled by the quality setting (1–100 in most editors). Quality 90 looks nearly identical to the original; quality 30 looks visibly degraded but the file is tiny.

How MP3 actually works (simplified):

  1. The audio is split into very short time chunks (a few milliseconds each).
  2. Each chunk is analysed for its frequency components.
  3. The MP3 encoder uses a psychoacoustic model — a mathematical model of human hearing. This model identifies sounds you definitely can't hear (e.g. a quiet flute playing at the same time as a loud trumpet — the flute is "masked").
  4. Inaudible/masked frequencies are removed. The remaining data is compressed.

This is why MP3 quality is measured in bitrate (kilobits per second). 128 kbps was the standard in the 2000s; 256–320 kbps is "high quality" and nearly indistinguishable from CD for most listeners.

🌍 Where you've used compression today (without realising) • Every photo you take with your phone is saved as JPEG automatically.
• Every song on Spotify is streamed as compressed audio (Ogg Vorbis or AAC).
• Every video on YouTube/Netflix uses H.264 or AV1 video codecs.
• WhatsApp / iMessage photos are heavily re-compressed before sending (which is why they look worse than the originals).
• Every web page sends compressed HTML/CSS/JS via Gzip or Brotli.

❌ Common student misconception "Compressing a file twice makes it even smaller." Usually false. After the first compression, the file is already close to its theoretical minimum size — there's no redundancy left to exploit. In some cases double-compression actually makes the file bigger (because of the algorithm's overhead). For lossy compression, repeated cycles also progressively destroy quality — this is what causes the "generation loss" you see when a meme is screenshotted, JPEG'd, screenshotted again, and reposted dozens of times.

💼 Industry insight — When compression goes wrong In 2017, Google's Pixel 2 phone applied aggressive JPEG compression to photos by default. 🌍 The future of compression — Machine learning New compression algorithms use neural networks trained on millions of images and sounds. 🤔 3.4.1 Why do we need encryption?

Data sent over the internet does not travel in a single private wire from your computer to a server. It hops across many routers and networks owned by different companies and governments. Anyone with the right equipment along the path can intercept the data — read it, copy it, or even modify it.

Encryption protects:

  • Personal data — passwords, addresses, dates of birth
  • Financial data — credit card numbers, bank logins, transaction details
  • Medical records — protected by law in most countries
  • Business data — trade secrets, customer lists, strategy
  • National security — diplomatic and military communication

Even if an attacker intercepts encrypted data, they see only the ciphertext. Without the key, recovering the plaintext would take longer than the age of the universe using modern ciphers.

🔒 HTTPS — encryption in action The little padlock in your browser's address bar means the page uses HTTPS (HTTP + SSL/TLS). Your connection is encrypted using asymmetric encryption to securely exchange a session key, then switches to fast symmetric encryption for the rest of the conversation. Without HTTPS, anyone on the same wi-fi network could read your messages and passwords.

People have been encrypting messages for over 2,000 years — long before computers existed. The four ciphers in the spec are all "paper-and-pen" ciphers that show how the core ideas of encryption developed. Modern ciphers (AES, RSA) work on the same principles but use astronomically larger keys and complex mathematical operations on binary data.

🔑 Symmetric

Problem: how do you give the key to the recipient without an attacker intercepting it? "The key distribution problem."

All four classical ciphers below are symmetric.
🔐🔑 Asymmetric (public-key)
Two mathematically-linked keys: public + private.

Public key shared with everyone; private key kept secret.

Anything encrypted with the public key can only be decrypted with the matching private key.

Solves the key-distribution problem. Used by HTTPS, PGP email, Bitcoin, digital signatures.
🐷 3.4.2a The Pigpen cipher

The Pigpen cipher (a.k.a. Freemason's cipher, Masonic cipher, tic-tac-toe cipher) is a substitution cipher — but unlike most substitution ciphers, it replaces each letter with a symbol rather than another letter.

The alphabet is laid out in two grids and two X-shapes (the second of each marked with dots). Each letter is replaced with the shape of the grid section surrounding it — including the dots if applicable.

To encrypt: for each plaintext letter, draw the shape of the surrounding grid section. For example, the letter A is in the top-left of the first 3×3 grid, so it's surrounded by lines on the right and bottom only — its symbol is "⌐" (a right angle opening to the top-left). The letter N (in the centre of the dotted grid) is surrounded on all sides plus has a dot.

🕯️ Historical note The Pigpen cipher was used by the Freemasons (a secret society) in the 18th century to protect their records. It also appeared on prisoner-of-war gravestones during the American Civil War. The game Assassin's Creed II uses a variant. Common alternative grids (shown in the textbook) rearrange the letters to make the cipher harder to crack.

Security: very weak. Because the same plaintext letter always maps to the same symbol, anyone who learns the grid can decrypt instantly. A determined cryptanalyst can also break unknown variants using frequency analysis — in English, the symbol for 'E' will appear most often, etc.

🏛️ 3.4.2b The Caesar cipher

Named after Julius Caesar (100–44 BC), who used it to send military orders. It's a substitution cipher in which every letter of the alphabet is shifted by the same number of places. The key is the shift value.

With a key of +3 (shift right by 3): A → D, B → E, C → F, … X → A, Y → B, Z → C. Letters near the end of the alphabet wrap around to the beginning.

PlaintextABC…WXYZ
Cipher (key +3)DEF…ZABC
📐 Worked example — encrypt with key +3 Plaintext: THIS IS A MESSAGE
Each letter +3: T→W, H→K, I→L, S→V, I→L, S→V, A→D, M→P, E→H, S→V, S→V, A→D, G→J, E→H
Ciphertext: WKLV LV D PHVVDJH (spaces and punctuation untouched)

Decryption: shift by the opposite amount. A key of +3 to encrypt → shift each ciphertext letter by −3 to decrypt.

🔧 Implementing the Caesar cipher in code

This is a perfect computational-thinking exercise. The Edexcel exam often asks you to plan or write a Caesar cipher algorithm. Use the ASCII codes from section 3.2.1:

  • Upper-case A–Z = ASCII 65–90
  • Lower-case a–z = ASCII 97–122

To encrypt a letter with shift k:

  1. Find its ASCII code.
  2. Add the shift.
  3. If the result > 90 (or 122 for lower case), subtract 26 to wrap around.
  4. If the result < 65 (or 97), add 26.
  5. Convert the new code back to a character.

Non-letters (spaces, commas, full stops, digits) should be left unchanged.

⚠️ Why the Caesar cipher is insecure

The Caesar cipher has only 25 possible non-trivial keys (shifts of 1 to 25). A computer can try all 25 in milliseconds — this is called a brute-force attack. The cipher is also vulnerable to frequency analysis: the most common letter in English text is 'E', so the most common letter in any Caesar ciphertext is probably the shifted version of E.

It's still useful as:

  • A teaching tool (clear demonstration of substitution and keys)
  • The building block of more advanced ciphers (e.g. Vigenère is essentially many Caesar ciphers stacked together)
  • The ROT-13 variant (key = 13) is still used online to hide spoilers in forum posts
📜 3.4.2c The Vigenère cipher

The Vigenère cipher (first described by Giovan Battista Bellaso in 1553, but later misattributed to Blaise de Vigenère) is a polyalphabetic substitution cipher — meaning it uses multiple Caesar ciphers with different shifts, cycling through them based on a keyword.

For centuries, the Vigenère cipher was believed to be unbreakable. It was nicknamed "le chiffre indéchiffrable" (the unbreakable cipher) and was even called "impossible of translation" by Scientific American in 1917. It was finally broken by Charles Babbage (the father of computing) in 1854 and independently by Friedrich Kasiski in 1863.

🛠️ How Vigenère works
  1. Choose a keyword (e.g. "code"). This is the encryption key.
  2. Write the keyword underneath the plaintext, repeating until the keyword sequence matches the length of the plaintext. The repeating sequence is called the keystream.
  3. For each plaintext letter, use the corresponding keystream letter as the Caesar shift — but expressed as "how far is this keystream letter from A?". Look up the result in the Vigenère grid.
📐 Worked example — encrypting "this is a message" with keyword "code"
Plaintextthisisamessage
Keystreamcodecodecodeco
CiphertextVVLWKWDQGGVWIS
For the first letter: plaintext 't', keystream 'c'. In the Vigenère grid, find row 't' (or column 't') and column 'c' (or row 'c'). They intersect at V. Repeat for every letter.

Decryption: use the keystream letter to find which column to look down. Find the ciphertext letter in that column. The plaintext letter is the row label.

💪 Why Vigenère is much stronger than Caesar
  • Same plaintext letter encrypts to different ciphertext letters at different positions (because the keystream letter changes). In the example above, 's' encrypts to W the first time and G the second time. This defeats simple frequency analysis.
  • The keyword can be any length — there are virtually unlimited keys. Brute force is infeasible by hand.

Weakness: the keyword repeats. Kasiski's attack works out the keyword length by finding repeating chunks in the ciphertext, then treats each "slice" as a separate Caesar cipher (which is easy to break).

🚂 3.4.2d The Rail Fence cipher

Unlike the previous three, the Rail Fence cipher is a transposition cipher — the letters of the plaintext are rearranged rather than substituted. Same letters, different order. The key is the number of "rails" (rows).

🛠️ How to encrypt with Rail Fence
  1. Decide the key — how many rails to use (typically 2–5).
  2. Write the plaintext in a zigzag pattern down and up across the rails, one letter per column.
  3. Read off the letters rail by rail, top to bottom, to get the ciphertext.
📐 Worked example — "This is a message" with key = 3 rails 
Rail 1:  t . . . i . . . e . . . g .
Rail 2:  . h . s . s . m . s . a . e
Rail 3:  . . i . . . a . . . s . . .

Read off rail 1: t i e g
Read off rail 2: h s s m s a e
Read off rail 3: i a s

Ciphertext: tieg hssmsae ias  →  TIEGHSSMSAEIAS
🔓 How to decrypt

Given the ciphertext and the key (number of rails), you need to rebuild the zigzag:

  1. Draw a grid with as many rows as the key, and as many columns as the ciphertext is long.
  2. Mark the zigzag pattern with dashes ("place-holders") in the cells the letters would occupy.
  3. Count how many dashes are on each rail.
  4. Fill in the dashes on rail 1 with the first batch of ciphertext letters (left-to-right). Then fill rail 2 with the next batch. Then rail 3 with the rest.
  5. Read the original message by following the zigzag pattern from left to right.
📐 Decryption walk-through — "TWLIIHSEBOGNWNENRSWNTEDAIIH" with key = 4 The ciphertext has 26 characters. Draw a 4-row grid with 26 dash positions following the zigzag pattern. Counting:
Rail 1: 5 dashes   ·   Rail 2: 9 dashes   ·   Rail 3: 9 dashes   ·   Rail 4: 3 dashes

Fill rail 1 with first 5 letters: T W L I I
Fill rail 2 with next 9 letters: H S E B O G N W N
Fill rail 3 with next 9 letters: E N R S W N T E D
Fill rail 4 with remaining 3 letters: A I I H (actually 4 — pattern dependent)

Reading the zigzag: THE ANSWER IS BLOWING IN THE WIND

Security: Rail Fence is weak. Only a small range of practical keys (typically 2–10 rails for sensibly-sized messages), so brute force trivially breaks it. It's also obvious that all the plaintext letters are present in the ciphertext, just shuffled — hints like letter frequencies are preserved.

🔬 Deep Dive — From classical ciphers to modern cryptography

The 2,000-year arms race. The story of encryption is essentially a battle between code-makers and code-breakers. Every cipher in this topic was considered "unbreakable" at some point in history:

  • Caesar (50 BC) — survived as a serious cipher for centuries, broken by Arab mathematician Al-Kindi around 800 AD using frequency analysis.
  • Vigenère (1553) — called "the unbreakable cipher" for 300 years, broken by Charles Babbage (1854) and independently by Friedrich Kasiski (1863).
  • Enigma machine (WW2) — Germany believed it was unbreakable, broken at Bletchley Park by Alan Turing and team using early computers. This shortened WW2 by an estimated 2 years.
  • DES (1976) — the US government's standard cipher, broken by brute force in 1998 (in 56 hours, by a $250,000 machine).
  • RSA-129 (1977) — predicted to take 40 quadrillion years to break. Actually broken in 1994 (17 years).

Modern ciphers like AES-256 are believed secure against any classical computer for billions of years. But quantum computers threaten this — see emerging trends.

💡 The "padlock" analogy for asymmetric encryption Imagine you want a friend to mail you a secret. You buy 1,000 identical padlocks (all open) and 1 key. You send the open padlocks to everyone — anyone can use one to lock a box. But only you have the key to open them. Anyone can encrypt (lock the padlock), only you can decrypt (open it with your private key). This is exactly how public-key cryptography works — your "public key" is the padlock; your "private key" is the only key that opens it. The clever maths is that you can give the padlock to everyone without revealing how the key works.
💡 The "secret alphabet" analogy for Caesar Picture a wheel with two rings of the alphabet, one inside the other. Spin the inner wheel by your key value (e.g. 3 places) and the rings are now offset. Look up any plaintext letter on the outer ring and read the cipher letter underneath on the inner ring. Hand a friend the same wheel and your shift key, and they can decode your messages. This actually existed in WW1 as the cipher disk — a portable mechanical Caesar cipher used by the US Army.

Why these classical ciphers fail. All four are vulnerable to specific attacks:

CipherMain weaknessHow it's broken
PigpenFixed substitution; only one mappingFrequency analysis (E is most common letter); pattern recognition once a few letters are known
CaesarOnly 25 keysBrute force: try all 25 shifts and see which gives English text

Frequency analysis — the most powerful attack on classical ciphers. In English, letter frequencies are roughly:

  • E (~12.7%), T (~9.1%), A (~8.2%), O (~7.5%), I (~7.0%), N (~6.7%)…
  • Rare: Q, X, Z, J (each <0.5%)

If you intercept a long Caesar ciphertext and notice that 'K' appears most often, you can guess K = E (shift +6). Decrypt with that shift and check if the result looks like English. Classical ciphers all fall to this kind of analysis given enough text.

🌍 Modern encryption you use every day • HTTPS (the padlock in your browser): uses TLS 1.3 with AES-256 and RSA/ECDHE.
• WhatsApp / Signal messages: end-to-end encrypted with the Signal Protocol (curve25519, AES-256). Not even Meta can read your texts.
• iPhone unlock / Face ID: hardware key in the Secure Enclave chip; Apple itself cannot retrieve your data.
• Wi-Fi (WPA3): AES-256 encrypts everything between your laptop and the router.
• Online banking: TLS + multi-factor authentication.
• Bitcoin: uses SHA-256 hashing and ECDSA digital signatures.
• Disk encryption (FileVault, BitLocker): AES-256 of your entire hard drive.
❌ Common student misconception "If a cipher uses really long keys, it's automatically unbreakable." Wrong. Vigenère can use a 50-character keyword (vastly more keys than Caesar), but the repetition of the key makes it crackable regardless. Security depends on the algorithm design, not just key length. A 256-bit AES key is secure; a 256-character Vigenère keyword is not. Modern cryptographers say "don't roll your own crypto" — even tiny design flaws can let attackers in.
⚙️ Pseudocode — Decrypt a Caesar cipher

Decryption is just encryption with the negative key.

Algorithm — Caesar decryption
# Decrypt a Caesar ciphertext (negative shift) RECEIVE ciphertext FROM (STRING) KEYBOARD RECEIVE shift FROM (INTEGER) KEYBOARD SET result TO "" FOR i FROM 0 TO LENGTH(ciphertext) - 1 DO SET code TO ord(ciphertext[i]) IF code >= 65 AND code <= 90 THEN # Note the +26 ensures the MOD works for negative shifts SET code TO ((code - 65 - shift + 26) MOD 26) + 65 ELSE IF code >= 97 AND code <= 122 THEN SET code TO ((code - 97 - shift + 26) MOD 26) + 97 END IF SET result TO result + chr(code) END FOR SEND result TO DISPLAY
⚙️ Pseudocode — Brute-force a Caesar cipher

This proves how weak Caesar really is — a computer can try every key in under a second.

Algorithm — Caesar brute force
# Try every possible Caesar shift and show all candidates RECEIVE ciphertext FROM (STRING) KEYBOARD FOR shift FROM 1 TO 25 DO SET candidate TO "" FOR i FROM 0 TO LENGTH(ciphertext) - 1 DO SET code TO ord(ciphertext[i]) IF code >= 65 AND code <= 90 THEN SET code TO ((code - 65 - shift + 26) MOD 26) + 65 END IF SET candidate TO candidate + chr(code) END FOR SEND "Shift " + shift + ": " + candidate TO DISPLAY END FOR # Human reads through the 25 candidates and picks the English one
⚙️ Pseudocode — Vigenère encryption
Algorithm — Vigenère cipher
# Encrypt with Vigenère using a keyword RECEIVE plaintext FROM (STRING) KEYBOARD RECEIVE keyword FROM (STRING) KEYBOARD SET keyword TO UPPER(keyword) SET result TO "" SET keyIndex TO 0 FOR i FROM 0 TO LENGTH(plaintext) - 1 DO SET code TO ord(UPPER(plaintext[i])) IF code >= 65 AND code <= 90 THEN # Calculate shift from current keyword letter SET shift TO ord(keyword[keyIndex MOD LENGTH(keyword)]) - 65 SET code TO ((code - 65 + shift) MOD 26) + 65 SET keyIndex TO keyIndex + 1 END IF SET result TO result + chr(code) END FOR SEND result TO DISPLAY

Trace through encrypting "HI" with keyword "AB":

  • i=0: code=72 ('H'), shift = ord('A')−65 = 0 → (72−65+0) MOD 26 = 7 → 7+65 = 72 ('H')
  • i=1: code=73 ('I'), shift = ord('B')−65 = 1 → (73−65+1) MOD 26 = 9 → 9+65 = 74 ('J')
  • Result: "HJ"
⚙️ Pseudocode — Rail Fence encryption
Algorithm — Rail Fence cipher
# Encrypt with Rail Fence cipher RECEIVE plaintext FROM (STRING) KEYBOARD RECEIVE rails FROM (INTEGER) KEYBOARD SET grid TO ARRAY of rails empty strings # Walk through plaintext placing letters on rails in zigzag pattern SET rail TO 0 SET direction TO 1 # 1 = going down, -1 = going up FOR i FROM 0 TO LENGTH(plaintext) - 1 DO SET grid[rail] TO grid[rail] + plaintext[i] # If we hit top or bottom rail, reverse direction IF rail = 0 THEN SET direction TO 1 ELSE IF rail = rails - 1 THEN SET direction TO -1 END IF SET rail TO rail + direction END FOR # Concatenate the rails to form ciphertext SET result TO "" FOR r FROM 0 TO rails - 1 DO SET result TO result + grid[r] END FOR SEND result TO DISPLAY
💼 Industry insight — The cryptography iceberg What you learn in this topic is just the tip. Modern cryptography is a vast field that includes:
• Hashing (one-way functions): SHA-256, used for password storage and blockchain.
• Digital signatures: prove who sent a message without revealing the message.
• Zero-knowledge proofs: prove you know a secret without revealing it.
• Homomorphic encryption: do calculations on encrypted data without decrypting first.
• Post-quantum cryptography: new ciphers (e.g. CRYSTALS-Kyber) resistant to quantum computers.
All build on the same fundamental ideas you're learning here — plaintext, ciphertext, keys, and the difference between symmetric and asymmetric.
