📜 Paper 2 — What you need to know
Paper 2 (4CP02) is a 3-hour on-screen practical exam, 80 marks, all questions mandatory. You write/debug/test Python 3 programs on a computer.

Common question types:
  • Complete a partial program (fill missing lines)
  • Fix syntax / logic / runtime errors in given code
  • Write a program from scratch using a structure provided
  • Use file handling (read CSV-style lines, save records)
  • Define functions (with/without parameters, with/without return)
  • Use ord() and chr() for ASCII encryption tasks
  • Process 1D and 2D lists (totals, averages, max/min, search)
  • Use random for games / simulation
  • Validate user input (range, length, type)

🛠 2.1 — Readable Code: Comments, Names & Indentation New▼

Good code is easy to read, maintain and debug. Examiners reward readable code with marks for "good style".

# GOOD: descriptive names, comments, indentation def calculate_average(total, count): # Guard against division by zero if count == 0: return 0 average = total / count # calculate mean return average # BAD: unclear names, no comments, hard to read def c(a, b): if b==0: return 0 return a/b
5 techniques for readable code:
• Comments — # explain WHY, not WHAT
• Descriptive names — student_age not x
• Indentation — 4 spaces per level (Python REQUIRES this)
• Constants — MAX_SIZE = 100 (UPPERCASE by convention)
• Subprograms — break code into named functions
✏️ Exam tip: "State two techniques to improve readability" is a frequent 2-mark question. Always mention comments AND descriptive names — you'll get the marks every time.

🔧 2.2 — Selection: if / elif / else New▼
# Standard if-elif-else (grade boundaries) score = int(input("Enter score: ")) ← cast str→int if score >= 90: ← checked FIRST grade = "A*" elif score >= 80: ← elif = "else if" grade = "A" elif score >= 70: grade = "B" elif score >= 50: grade = "C" else: ← catches everything else grade = "U" print(f"Grade: {grade}")
⚠️ Common slip: Using = (assign) instead of == (compare) inside the condition. if x = 5: → SyntaxError. The correct form is if x == 5:

🔁 2.2 — Iteration: for and while loops (deep dive) New▼
FOR loop — known repetitions
# range(start, stop, step) — stop EXCLUDED for i in range(1, 6): ← 1,2,3,4,5 print(i) for i in range(10): ← 0..9 print(i) for i in range(2, 11, 2): ← 2,4,6,8,10 print(i) # Iterate over a list names = ["Ali", "Bea", "Cal"] for name in names: ← gets each value print(name) # Iterate over a string (char by char) for letter in "HELLO": print(letter)
WHILE loop — condition-controlled
# Repeats UNTIL condition is False password = "" while password != "secret": ← keep asking password = input("Password: ") print("Access granted!") # Counting with while count = 0 while count < 5: print(count) count += 1 ← MUST update! # Validation pattern (very common) age = -1 while age < 0 or age > 120: age = int(input("Age: "))
⚠️ Infinite loop trap: If you forget to update the variable that controls the while condition, the loop runs forever. Always make sure something CHANGES inside the loop.
✏️ When to use which: Use for when you know the count (5 students, 10 scores). Use while when you don't know in advance (validation, "until done").
📦 2.3 — Data Types: int, float, bool, str + Casting New▼
# The four basic types age = 16 # int — whole numbers height = 1.75 # float — decimal numbers passed = True # bool — True or False (CAPITAL T/F!) name = "Alice" # str — text, in quotes # Check the type print(type(age)) # <class 'int'> print(type(height)) # <class 'float'>
🔑 The single most important rule on Paper 2:
input() ALWAYS returns a string. You MUST convert with int() or float() before doing arithmetic, or your code will crash or behave wrong.
# ── CASTING (Type Conversion) ── # String → Integer (str → int) text = "42" number = int(text) # 42 (now an int) print(number + 8) # 50 (works!) # String → Float (str → float) price = float("19.99") # 19.99 # Integer → String (int → str) needed for concatenation score = 95 print("Score: " + str(score)) # works # print("Score: " + score) # TypeError! # Reading numbers from user — the standard pattern age = int(input("Age: ")) # nested calls print(age + 1) # works because age is int

⚠️ Top mistake: a = input("a:"); b = input("b:"); print(a + b). If user types 3 and 4, output is "34" not 7! Strings concatenate; you wanted ints.
🔡 2.3 — String, ord() & chr() — ASCII codes (Paper 2 favourite) New▼
Every character has an ASCII number: A=65, B=66, ..., Z=90 · a=97, b=98, ..., z=122 · 0=48, ..., 9=57 · space=32.
Paper 2 LOVES encryption questions using these.
# ord(char) → ASCII number · chr(num) → character print(ord("A")) # 65 print(ord("a")) # 97 print(ord("0")) # 48 print(chr(65)) # "A" print(chr(97)) # "a" print(chr(66)) # "B" # ── A Caesar-cipher (shift by 3) — classic exam Q ── plaintext = "HELLO" cipher = "" for letter in plaintext: code = ord(letter) # letter → ASCII number new_code = code + 3 # shift new_letter = chr(new_code) # back to letter cipher = cipher + new_letter print(cipher) # "KHOOR"
# String slicing & indexing reminder name = "Python" print(name[0]) # "P" ← first char print(name[-1]) # "n" ← last char print(name[0:3]) # "Pyt" ← slice 0,1,2 (end EXCLUDED) print(len(name)) # 6 print(name.upper()) # "PYTHON" print(name.lower()) # "python" print(name + "3") # "Python3" ← concatenation print("y" in name) # True ← membership test
✏️ Exam tip: If asked to write a "shift cipher" or "encrypt this word", the algorithm is ALWAYS: loop → ord() → add shift → chr() → concatenate. Memorise this pattern.
📋 2.3 — 1D Lists: create, access, modify, traverse New▼
# Create a 1D list (array) scores = [85, 72, 90, 61, 78] ← 5 elements, indexes 0-4 # ── ACCESS by index ── print(scores[0]) # 85 ← first (index 0) print(scores[2]) # 90 ← third print(scores[-1]) # 78 ← last (negative from end) print(len(scores)) # 5 # ── MODIFY ── scores[2] = 95 # change index 2 to 95 scores.append(88) # add to END → [85,72,95,61,78,88] scores.remove(61) # remove by VALUE → [85,72,95,78,88] scores.insert(0, 100) # insert at index 0 → [100,85,72,95,78,88] scores.pop() # remove & return LAST # ── TRAVERSE (loop through) ── for score in scores: # each VALUE in turn print(score) for i in range(len(scores)): # by INDEX (when you need it) print(i, scores[i])
# ── Common Paper 2 list operations ── nums = [12, 45, 7, 23, 8, 91, 3] print(sum(nums)) # 189 — total of all elements print(sum(nums) / len(nums)) # 27.0 — average print(max(nums)) # 91 — largest print(min(nums)) # 3 — smallest print(len(nums)) # 7 — number of elements print(45 in nums) # True — is 45 a member? nums.sort() # sort in place print(nums) # [3, 7, 8, 12, 23, 45, 91] # ── Find largest WITHOUT using max() — exam favourite ── biggest = nums[0] # start with first element for n in nums: # check every other one if n > biggest: biggest = n print(biggest) # 91 # ── Count items matching a condition ── high_count = 0 for score in nums: if score >= 30: high_count = high_count + 1 print(high_count) # 2 (45 and 91)
✏️ Exam tip: Be fluent in BOTH "loop and accumulate" (no built-ins) AND using sum/max/min/len. Some questions explicitly say "without using built-in functions" — be ready for both.
🔲 2.3 — 2D Lists: rows, columns, nested loops New▼
A 2D list is a list whose elements are themselves lists. Think of it as a grid / table: outer list = rows, inner lists = the values in each row. Access with TWO indexes: grid[row][col].
# ── A 3x3 grid (rows of numbers) ── grid = [ [1, 2, 3], # row 0 [4, 5, 6], # row 1 [7, 8, 9] # row 2 ] print(grid[0]) # [1, 2, 3] — whole first row print(grid[1][2]) # 6 — row 1, column 2 print(grid[2][0]) # 7 — row 2, column 0 grid[0][0] = 99 # modify a single cell # ── Traverse the WHOLE grid with nested for loops ── for row in grid: for value in row: print(value, end=" ") print() # newline after each row
# ── Records as 2D list (very common in Paper 2) ── # Each inner list = [name, age, score] students = [ ["Alice", 16, 85], ["Bob", 15, 72], ["Cara", 17, 91] ] # Print each student's name and score for student in students: print(student[0], student[2]) # Average score (col index 2) — accumulator pattern total = 0 for student in students: total = total + student[2] # add the score column average = total / len(students) print("Average:", average) # 82.6...

📁 2.4 — File Handling: text, numbers, comma-separated values New▼
Files are tested every year on Paper 2. Know the 3 modes, the 4 verbs (open/read/write/close), and the CSV-style pattern below.
ModeMeaningIf file existsIf file missing
"r"ReadOpens itFileNotFoundError
"w"Write (overwrite)Wipes content!Creates new file
"a"AppendAdds to endCreates new file
# ── WRITING strings to a file ── file = open("scores.txt", "w") # "w" — fresh file file.write("Alice: 95\n") # \n = newline file.write("Bob: 78\n") file.close() # ALWAYS close! # ── APPENDING (add without deleting) ── file = open("scores.txt", "a") file.write("Carol: 82\n") file.close() # ── WRITING numbers (must cast to str first!) ── score = 85 file = open("data.txt", "a") file.write(str(score) + "\n") # int → str + newline file.close()
# ── READING a file (3 ways) ── # Way 1 — read EVERYTHING into one string file = open("scores.txt", "r") content = file.read() file.close() print(content) # Way 2 — line by line with a for loop (best for big files) file = open("scores.txt", "r") for line in file: print(line.strip()) # .strip() removes \n file.close() # Way 3 — read all lines into a LIST file = open("scores.txt", "r") lines = file.readlines() # list of "Alice: 95\n", ... file.close()
# ── CSV-style files (comma-separated) — VERY common ── # File content (students.txt): # Alice,16,85 # Bob,15,72 # Cara,17,91 # ── WRITING comma-separated records ── file = open("students.txt", "w") file.write("Alice,16,85\n") file.write("Bob,15,72\n") file.close() # ── READING and SPLITTING by comma ── file = open("students.txt", "r") for line in file: line = line.strip() # remove \n parts = line.split(",") # ["Alice","16","85"] name = parts[0] age = int(parts[1]) # cast — file fields are strings! score = int(parts[2]) print(name, "is", age, "and scored", score) file.close()
⚠️ Critical: Everything read from a file is a STRING. To do arithmetic on file data (sum scores, etc.) you MUST cast: int(parts[2]) or float(...).
✏️ Exam tip: Best practice uses with open(...) as f: which auto-closes. Both styles are accepted on Paper 2 — but if you use plain open(), you MUST remember .close().
🎲 2.4 — Random module: randint, choice, random New▼
# ── ALWAYS import random first ── import random # randint(a, b) — integer from a to b INCLUSIVE on both ends dice = random.randint(1, 6) # 1, 2, 3, 4, 5 or 6 lottery = random.randint(1, 49) # 1..49 inclusive # choice(list) — pick one random item from a list suits = ["♠", "♥", "♦", "♣"] suit = random.choice(suits) # one of the 4 # random() — float between 0.0 and 1.0 chance = random.random() # e.g. 0.7234
# ── Number-guessing game (classic Paper 2 task) ── import random target = random.randint(1, 100) guess = 0 attempts = 0 while guess != target: guess = int(input("Guess (1-100): ")) attempts = attempts + 1 if guess < target: print("Too low!") elif guess > target: print("Too high!") print(f"Correct in {attempts} attempts!")
⚠️ Easy mark loss: Forgetting import random at the top of the program. It's worth a mark on its own in many tasks.
✏️ Note: randint(1, 6) includes BOTH endpoints — different from range(1, 6) which excludes 6. Easy to confuse — read carefully.
➕ 2.5 — Arithmetic, Relational & Logical Operators New▼
# ── ARITHMETIC operators ── a, b = 17, 5 print(a + b) # 22 — addition print(a - b) # 12 — subtraction print(a * b) # 85 — multiplication print(a / b) # 3.4 — division (ALWAYS float) print(a // b) # 3 — floor/integer division print(a % b) # 2 — modulus (remainder) print(a ** b) # 1419857 — exponentiation (a to the power b) # ── RELATIONAL operators (return True/False) ── print(a == b) # False — equal to (TWO equals signs!) print(a != b) # True — not equal print(a > b) # True — greater than print(a < b) # False print(a >= b) # True — greater or equal print(a <= b) # False # ── LOGICAL operators (combine conditions) ── age = 17 if age >= 13 and age <= 19: # BOTH must be True print("Teenager") if age < 5 or age >= 65: # EITHER True print("Free ticket!") if not (age >= 18): # reverses True/False print("Under 18")
# ── % (modulus) is HEAVILY tested — memorise these ── print(10 % 2) # 0 — 10 is EVEN print(7 % 2) # 1 — 7 is ODD if n % 2 == 0: # test for even print("Even") print(345 % 10) # 5 — LAST digit of 345 print(345 // 10) # 34 — REMOVE last digit if year % 4 == 0: # leap year (simplified) print("Leap year")
✏️ Exam tip: % applications you MUST know: even/odd test (n%2==0), getting last digit (n%10), removing last digit (n//10), every-nth-iteration (i%5==0).
🔧 2.6 — User-defined Functions: 3 variants you MUST know New▼
Paper 2 will ask you to write/identify ALL THREE variants: (1) function with parameters & return, (2) function with parameters but no return (procedure), (3) function with NO parameters.
# ════════════════════════════════════════════════════ # VARIANT 1: WITH parameters, WITH return (most common) # ════════════════════════════════════════════════════ def calculate_area(width, height): ← 2 parameters area = width * height return area ← sends value BACK # Call it — store the returned value result = calculate_area(5, 3) # 5 and 3 are arguments print(result) # 15 print(calculate_area(10, 4)) # can use directly: 40
# ════════════════════════════════════════════════════ # VARIANT 2: WITH parameters, NO return (a "procedure") # ════════════════════════════════════════════════════ def print_receipt(item, price): ← parameters, but… print(f"Item: {item}") print(f"Price: £{price}") print("-" * 20) # no return statement → procedure # Call it — don't assign because nothing comes back print_receipt("Bread", 1.20) print_receipt("Milk", 0.95)
# ════════════════════════════════════════════════════ # VARIANT 3: NO parameters — useful for menus / banners # ════════════════════════════════════════════════════ def show_menu(): ← empty brackets print("1. New game") print("2. Load game") print("3. Quit") def welcome(): name = input("Your name? ") # input happens inside print(f"Hello, {name}!") return name # can still return! # Call them — empty brackets! show_menu() player = welcome() print(player)
# ── Vocabulary: parameter vs argument ── def greet(name): # 'name' is a PARAMETER print("Hi,", name) greet("Alice") # "Alice" is an ARGUMENT
Local vs Global scope
total = 100 # GLOBAL — accessible everywhere def add_score(points): global total # tell Python: use the global one local_var = points # LOCAL — dies when function ends total += points return total add_score(25) # total is now 125 # print(local_var) # NameError — local_var doesn't exist here!

Q1 — Readability Easy · 2.1 Style of: Specimen Q1(b) 4 marks

State two techniques a programmer can use to improve the readability of their code. For each technique, give an example.

✅ Mark Scheme (any 2 techniques × 2 marks)
  • Comments — lines starting with # explaining the code; e.g. # calculate total price [1+1]
  • Descriptive variable names — meaningful names; e.g. student_age instead of x [1+1]
  • Indentation — consistent 4-space indentation showing structure [1+1]
  • Subprograms / Functions — breaking code into named functions [1+1]
  • Constants — meaningful constant names like MAX_SCORE = 100 [1+1]
💡 WHY each technique works
  • Comments describe the intent behind code that the syntax alone can't show. A line like x = x * 1.2 tells you nothing; # add 20% VAT tells you everything. Examiners reward comments that explain WHY, not what.
  • Descriptive names remove the need for guessing. final_price vs fp — the first one reads like English in every line that uses it. Use snake_case (lowercase with underscores) for variables, ALL_CAPS for constants.
  • Indentation in Python isn't decoration — it defines what's inside an if or for block. Consistent 4-space indentation makes nesting obvious at a glance. Mixing tabs and spaces causes IndentationError.
  • Subprograms follow the DRY principle (Don't Repeat Yourself). One named function like calculate_average(numbers) can be re-used many times, tested once, and read aloud easily.
  • Constants name "magic numbers" so a value's meaning is visible. if score > PASS_MARK reads better than if score > 50, and changing the rule means editing one line.
🎯 Exam tip: The question asks for two techniques with an example. Don't write three. Write two, and make sure each example is a single concrete line of Python — not a generic phrase like "use better names".

✅ Mark Scheme — Model answer

Also accept: while True: with break, or a direct check using any(c.isdigit() for c in password).

💡 WHY each line earns its mark
  • Helper function has_digit(): a function that returns True or False is called a predicate. It hides the "loop through every character" logic behind one readable name. Notice it loops, and the FIRST time it sees a digit, return True exits the function early — efficient, because we don't keep checking once we know.
  • ch.isdigit(): a built-in string method that returns True if the single character is 0–9, otherwise False. Not ch == "digit". Built-in methods are cleaner and harder to get wrong than home-made checks.
  • password = "" BEFORE the loop: we must initialise the variable so the while condition has something to evaluate on the first check. "" has length 0 (less than 8) so the loop is guaranteed to run at least once — exactly what we want.
  • Choosing while over for: a for loop repeats a known number of times. Here we don't know how many guesses the user will need — could be one, could be ten. while repeats until a condition becomes false. Rule of thumb: known count → for; unknown count → while.

The or means the loop continues as long as any rule is broken. Only when BOTH rules pass (length ≥ 8 AND has a digit) do both sides become false, and the loop exits.
  • input() inside the loop: if you put the input() outside the loop, you'd only ask once and then loop forever on the bad value (an infinite loop). The input must be inside so each iteration prompts again.
  • if / elif for the error messages: these are diagnostic — they tell the user which rule they broke. Without them, the user sees no feedback and may keep typing the same bad password. Examiners reward "user-friendly" output.
  • The final print is OUTSIDE the loop: notice it's not indented under while. That's intentional — it only runs ONCE, after the loop ends successfully. Putting it inside would print "Password accepted!" on every wrong try.
🎯 Exam tip: When you see "keep asking until…", that's a while loop trigger every time. Plan three things on rough paper before coding: (1) what's the EXIT condition? (2) what to do INSIDE the loop? (3) how does the loop variable CHANGE so we eventually exit? Forgetting #3 causes infinite loops.

Write the three subprograms and test code that adds three scores and prints the average.

Q6 — Reading a CSV-style file Hard · 2.4 Style of: 2022 Q3 7 marks
A file called orders.txt contains book orders, one per line, in this format:
Alice,Maths,3
Bob,Physics,1
Cara,Maths,5
Dave,English,2

Write a Python program that reads the file and prints (a) the total number of books ordered, and (b) the name and quantity of the customer who ordered the most books.

✅ Mark Scheme (7 marks)
file = open("orders.txt", "r") # [1] open in read mode total = 0 biggest_qty = 0 biggest_name = "" for line in file: # [1] iterate lines line = line.strip() # [1] strip \n parts = line.split(",") # [1] split on comma name = parts[0] quantity = int(parts[2]) # [1] cast to int total = total + quantity # [1] accumulate if quantity > biggest_qty: # [1] track max biggest_qty = quantity biggest_name = name file.close() # [1] close file print("Total books:", total) print("Biggest order:", biggest_name, biggest_qty)
💡 WHY each step — file handling decoded line by line
  • open("orders.txt", "r") — what the modes mean:
    • "r" = READ. Opens an existing file for reading. Crashes if the file doesn't exist.
    • "w" = WRITE. Creates a new file, OR overwrites and erases an existing one. Dangerous if you have valuable data — it deletes first, asks questions never.
    • "a" = APPEND. Adds new content to the end of an existing file, keeping the old data.
  • Why for line in file:? Python is clever — it gives you ONE line at a time, including the invisible newline character (\n) at the end of each. You don't have to know how many lines exist. The loop just stops when the file ends.
  • WHY .strip()? The newline character \n sits at the end of every line read from a file. If your line is "Amy,books,5\n" and you split it, the last part becomes "5\n" — and int("5\n") can fail or produce subtle bugs. .strip() removes all whitespace (spaces, tabs, AND newlines) from BOTH ends. After line.strip(), the line is just "Amy,books,5" — clean. Forget .strip() and you'll lose marks AND get runtime errors.
  • WHY .split(",")? A CSV line is one big string with commas as separators. .split(",") chops it into a LIST of separate strings — one item per field. The comma is the separator, not part of the data. Example: "Amy,books,5" becomes ["Amy", "books", "5"]. You can now access each field with parts[0], parts[1], parts[2] — like cells in a spreadsheet row.
  • WHY cast with int(parts[2])? Everything from a file is a STRING. parts[2] is "5", not 5. If you do total + parts[2], Python will refuse — you can't ADD a string to a number. int() converts the string to an actual integer so arithmetic works.
  • Accumulator pattern: total = total + quantity. The variable total starts at 0 BEFORE the loop. Each iteration adds the new quantity to it. By the end, total holds the sum of every quantity in the file. This is the standard "running total" pattern — memorise it.
  • "Track the max" pattern: biggest_qty = 0 outside the loop (initialise to a value lower than any possible data). Inside, if the current value is BIGGER than what we've seen, update both the max value AND the matching name. By the end, you have the winner. Used in EVERY paper.
  • WHY file.close()? Files held open lock system resources. On Windows, you can't even delete an open file. Always close. Better: use with open(...) as file: — Python closes it automatically when the block ends, even if your code crashes. Most modern mark schemes accept either, but with is the safer style.
🎯 Exam tip: The pattern line.strip().split(",") appears in nearly every IGCSE file question. Burn it into muscle memory. After splitting, IMMEDIATELY cast any numeric fields with int() or float() — don't leave it until you need them, because that's when you forget.
Q7 — ord() and chr() encryption Hard · 2.3 Style of: 2022 Q4 6 marks
A Caesar cipher shifts each letter by a fixed amount. For shift = 3, A→D, B→E, ..., HELLO → KHOOR. ASCII: A=65, Z=90.

Write a Python program that asks the user for a plaintext word (in uppercase, A–Z only) and a shift number, then prints the encrypted ciphertext. Use ord() and chr().

✅ Mark Scheme (6 marks)
plaintext = input("Plaintext: ") # [1] get input shift = int(input("Shift: ")) # [1] cast shift cipher = "" # [1] init empty string for letter in plaintext: # [1] loop chars code = ord(letter) # [1] ord new_code = code + shift new_letter = chr(new_code) # [1] chr cipher = cipher + new_letter print(cipher)

Award an extra mark for wrap-around with ((code - 65 + shift) % 26) + 65, though this is not required for full marks unless asked.

💡 WHY ord, chr, and the loop work this way
  • ASCII — the bridge between letters and numbers: every character on your keyboard has a number called its ASCII code. "A" is 65, "B" is 66, ... "Z" is 90. Lowercase: "a" is 97 ... "z" is 122. The gap between lowercase and uppercase is exactly 32. Encryption needs numbers, not letters — so we use ASCII.
  • ord(letter) = LETTER → NUMBER. Stands for "ordinal". Give it ONE character, it gives back the ASCII code. ord("A") = 65. ord("z") = 122. Cannot be passed a multi-character string.
  • chr(code) = NUMBER → LETTER. Stands for "character". The inverse of ord. chr(65) = "A". chr(97) = "a". chr(33) = "!". Always returns a 1-character string.
  • WHY cipher = "" first? We need a place to BUILD UP the answer one letter at a time. Starting empty, each loop iteration adds one encrypted letter using cipher = cipher + new_letter. This is the string-building accumulator pattern — same shape as a number total, but with strings.
  • WHY loop with for letter in plaintext? Python lets you iterate over a string directly — each iteration letter holds ONE character. If plaintext is "CAT", the loop runs 3 times: letter = "C", then "A", then "T". No index math needed.
  • The shift trick: Add a fixed number to every ASCII code. With shift=3, "A" (65) becomes 68 → chr(68) = "D". "B" → "E". "C" → "F". Caesar cipher, used 2000 years ago. Anyone knowing the shift can REVERSE it: chr(code - shift).
  • WHY use %26 for wrap-around? Without it, encrypting "Z" with shift=3 gives chr(93) = "]" (a bracket, not a letter!). The %26 wraps you back to the start of the alphabet so "Z"+3 = "C". The full formula ((code - 65 + shift) % 26) + 65 works as follows: subtract 65 to scale "A"=0 ... "Z"=25, add the shift, do mod 26 to wrap, then add 65 back to return to ASCII range. Memorise this — it appears in EVERY cipher question.
  • What if the user types lowercase? The formula above assumes uppercase. For mixed case, either call .upper() on the input first, OR check each letter's range and use 65 vs 97 accordingly. The exam usually specifies "uppercase only" to keep it simple — read the question.
🎯 Exam tip: If you see "encryption" or "cipher" or "shift the letters", reach for ord() and chr() immediately. They go together — one converts in, the other converts out. Practise the wrap-around formula until you can write it without thinking.

💡 WHY 2D lists work like this
  • A 2D list is a list-of-lists. Each "row" is itself a list. students here is a list with 4 items; each of those 4 items is a 3-element list (the record). Think of it as a small spreadsheet with rows and columns — rows = records, columns = fields.

Without return, the caller would get None — the function would work internally but its result would be lost. Q10 — Linear search in a list Medium · 2.3 + 2.6 Style of: 2023 Q2 6 marks

Write a function find_index(my_list, target) that searches for target in my_list and returns the index where it is found, or -1 if not found. Then demonstrate it on [10, 20, 30, 40] with target = 30 and with target = 99.

✅ Mark Scheme (6 marks)
def find_index(my_list, target): # [1] def with 2 params for i in range(len(my_list)): # [1] loop indices if my_list[i] == target: # [1] compare return i # [1] return index on hit return -1 # [1] not found print(find_index([10, 20, 30, 40], 30)) # 2 print(find_index([10, 20, 30, 40], 99)) # -1 [1] both calls correct
💡 WHY the linear search works this way
  • What "linear search" means: we check items ONE BY ONE from the start of the list until we find the target — or run out. It's the simplest search algorithm. Other algorithms (like binary search) are faster but require a SORTED list. Linear search works on ANY list, sorted or not — and is the only kind IGCSE expects.
  • WHY for i in range(len(my_list)): instead of for item in my_list:? The question asks us to return the INDEX (the position) where we found the target, not the value itself. To know the position, we have to track i as it counts up. range(len(my_list)) gives us 0, 1, 2, ..., up to (length − 1) — exactly the valid indices for the list.
  • len(my_list) gives the number of items in the list. For [10, 20, 30, 40], len returns 4, so range(4) yields 0, 1, 2, 3 — every valid index. Going to 4 itself would crash with IndexError: list index out of range.
  • my_list[i] == target — the comparison: my_list[i] means "the item at position i". == is comparison (not =, which is assignment). On a hit, the if-block runs. On a miss, the loop moves on to the next i.
  • return i INSIDE the loop: the moment we find a match, we return the index IMMEDIATELY. return not only sends a value back — it also EXITS the function instantly. The loop stops. No further checks happen. This is efficient: why keep looking once we've found it?
  • return -1 AFTER the loop: if the loop runs through every item without returning, control falls out the bottom of the loop. return -1 tells the caller "not found". Why -1? Because -1 is NEVER a valid list index (well, technically it means "last item" in Python — but as a search RESULT, it's a clear sentinel meaning "didn't find it"). Examiners accept -1, None, or False as the "not found" signal.
  • WHY pass my_list as a parameter (instead of using a global)? Parameters make the function REUSABLE — it can search ANY list, not just one hard-coded list. The same find_index works on [10,20,30,40] and on [3,7,1,9,5,2] without changing the function. That's the whole point of subprograms.
  • Two demonstration calls — found and not-found: find_index([10,20,30,40], 30) returns 2 (index of 30). find_index([10,20,30,40], 99) returns -1 (not in list). Examiners want to see BOTH cases tested. A function that only works for the "happy path" is incomplete.
  • What if there are DUPLICATES? This version returns the index of the FIRST match only (because of return i in the loop). If the list were [10,30,30,40] with target 30, you'd get index 1, not index 2. That's usually the right behaviour — but read the question carefully; some variants ask for all matches.
🎯 Exam tip: Search functions ALWAYS return either an index (or value) on hit, AND a sentinel (-1, None, "") on miss. Plan both paths before coding. The "not found" return goes AFTER the loop, not inside an else branch on the if — that's the most common student mistake.
Q11 — Save records to file, then total Hard · 2.4 Style of: 2024 Q3 9 marks

Write a Python program that:
(a) Asks the user for product name and price (validate price 0–1000) for as many products as they want — until they type "done" as the name.
(b) Saves each valid record to a file products.txt in the format name,price on each line.
(c) Reopens the file and prints the TOTAL price of all products.

✅ Mark Scheme (9 marks)
file = open("products.txt", "w") # [1] open for write while True: # [1] loop until "done" name = input("Product (or 'done'): ") if name == "done": # [1] exit condition break price = float(input("Price: ")) # [1] cast while price < 0 or price > 1000: # [1] validate print("Invalid — try 0..1000") price = float(input("Price: ")) file.write(name + "," + str(price) + "\n") # [1] write CSV line file.close() # [1] close # Read back and total file = open("products.txt", "r") # [1] reopen for read total = 0 for line in file: parts = line.strip().split(",") total = total + float(parts[1]) # [1] accumulate file.close() print("Total: £", total)
💡 WHY each step — file write THEN file read explained
  • This is a TWO-PHASE program — write phase then read phase. First we collect data from the user and SAVE it to a file. Then we close the file, REOPEN it for reading, and process the saved data. This pattern (save → reopen → process) appears in nearly every higher-mark file question.
  • open("products.txt", "w") — what "w" actually does: "w" = WRITE mode. It creates the file if it doesn't exist, OR completely erases the existing file before writing. If products.txt had old data, it's GONE the moment you open in "w". Use "a" (append) if you want to keep existing data and add to it.
  • while True: + break: the "infinite loop with manual exit": sometimes you don't know your exit condition at the top of the loop — it depends on user input INSIDE the loop. while True means "loop forever". The break statement is your escape hatch: when the user types "done", we leave the loop. Without break, this would never end.
  • if name == "done": break BEFORE the price prompt: the exit check has to come BEFORE we ask for a price. Otherwise typing "done" would still trigger an unnecessary float(input()) which would crash because "done" isn't a number. Order matters: check for exit FIRST, then process valid input.
  • WHY float(input(...)) for price? Prices have decimals (£1.50, £2.99). int() would refuse "1.50" and crash. float() accepts both whole numbers and decimals. Always pick the cast that matches your data — int for counts/ages, float for measurements/money.
  • The INNER while-loop is VALIDATION: while price < 0 or price > 1000: re-prompts until the user gives a valid price. This is a NESTED loop — a while inside a while. The OUTER loop handles the product list; the INNER loop handles validation of one price. Indentation makes the structure clear.
  • file.write(name + "," + str(price) + "\n") — building a CSV line: three things to notice. (1) The COMMA goes between fields — it's our separator. (2) str(price) converts the float to a string because you can only concatenate strings, not mix types. (3) The \n at the end is the NEWLINE character — without it, every product would be written on the SAME line and you couldn't read them back line by line.
  • WHY file.close() before reopening? A file opened in "w" mode buffers your writes — the actual disk write may not finish until you close. If you try to reopen it for reading without closing first, you might see EMPTY or PARTIAL data. close() flushes the buffer and releases the file lock. (The modern with open(...) as f: does this automatically — and is preferred.)
  • Reopening with open("products.txt", "r"): a fresh handle, this time in READ mode. Reading is non-destructive — you can read the same file repeatedly. Different handle, same file on disk.
  • WHY line.strip().split(",")? Two surgical steps. .strip() removes the trailing \n newline character that for line in file includes on every line — without this, your last field would still have a \n stuck to it and float("1.50\n") behaves unpredictably. .split(",") chops the cleaned line at every comma, giving a list of fields. CHAINED in one line — outer-to-inner: strip first, then split the stripped result.
  • total = total + float(parts[1]) — the read-back accumulator: parts is the list [name, price_as_string]. parts[1] is the price field (still a string). float() casts it back to a number so we can add. total was initialised to 0 above the loop; each iteration adds one product's price to it.
  • Two file handles, two closes: we wrote, closed; we read, closed. Each open() needs its matching close(). Forgetting one is a common slip — examiners specifically look for both.
🎯 Exam tip: The phrase "save the records to a file, THEN read them back" should trigger this exact two-phase template in your head: (1) open "w", loop+write, close — (2) open "r", loop+process, close. Memorise the skeleton; fill in the details from the question. Use "a" (append) instead of "w" if the question says "add to existing file".
Q12 — Full Paper-2-style mini exam Hard · ALL Mixed style 12 marks

Write a complete Python program for a tuck-shop:

  • A function show_menu() with no parameters prints a menu of 3 items + prices.
  • A function get_choice() uses validation (must be 1, 2 or 3) and returns the choice.
  • A function price_of(choice) takes the choice and returns the price.
  • The main program calls them in a loop. It keeps a running total. When the user enters 0, it prints the total and ends.

✅ Mark Scheme (12 marks)
def show_menu(): # [1] no-param procedure print("1. Chips £1.50") # [1] correct output print("2. Drink £1.00") print("3. Bar £0.80") print("0. Finish") def get_choice(): # [1] returns value choice = -1 while choice < 0 or choice > 3: # [1] validation choice = int(input("Choice: ")) # [1] cast input return choice # [1] return def price_of(choice): # [1] param + return if choice == 1: return 1.50 # [1] correct prices if choice == 2: return 1.00 if choice == 3: return 0.80 return 0 # ── Main program ── total = 0 # [1] accumulator while True: # [1] loop show_menu() choice = get_choice() if choice == 0: # [1] exit break total = total + price_of(choice) # [1] add price print(f"Total: £{total:.2f}") # [1] print total
💡 WHY this program ties together the WHOLE syllabus
  • This is a "full program" question — and they're all built the same way: a small set of functions on top, a main program at the bottom that calls them in a loop. Examiners design these to test EVERY skill in one go: subprograms, parameters, return values, validation, loops, conditional logic, accumulators, and formatted output.
  • show_menu() — no parameters, no return: a "procedure" that does a fixed job — printing the menu. It uses neither input nor output (other than the print side-effect). Called multiple times from the main loop. Reusable: if the menu changes, you edit ONE function and every call updates automatically.
  • get_choice() — validation pattern bundled into a function: choice = -1 initialises with a value GUARANTEED to fail the validation check, so the loop runs at least once. while choice < 0 or choice > 3: means "keep asking until they give 0, 1, 2, or 3". return choice hands the validated number back to the main program. This wraps a 4-line validation pattern into a one-line call: choice = get_choice().
  • WHY int(input(...)) inside get_choice? Same trap as always — input() returns a string. Without int(), the comparison choice < 0 would raise TypeError. Cast IMMEDIATELY on read.
  • price_of(choice) — parameter AND return: takes a number, returns a number. The pattern of if x == 1: return 1.50 uses an EARLY return — as soon as we find the matching choice, we return the price. Equivalent to if/elif/else; mark schemes accept either style. The final return 0 handles "unknown choice" so the function ALWAYS returns something (never None by accident).
  • WHY the main program uses while True: ... break: we don't know how many items the customer will buy. while True with an if choice == 0: break is the standard "loop until sentinel value" pattern. The exit condition lives INSIDE the loop because it depends on user input.
  • total = 0 BEFORE the loop, then total = total + price_of(choice): the running total accumulator. Initialise outside (because the loop body needs to read its existing value), then add each item's price inside. By the end, total holds the sum of all prices.
  • The function CALL inside an expression: total + price_of(choice) calls price_of, takes the returned price, and adds it to total. The function call IS its return value once it finishes. This composability is the whole reason we use returning functions.
  • WHY check for 0 AFTER calling get_choice but BEFORE adding? If we didn't, choosing 0 (exit) would still attempt price_of(0) and add 0 to the total — wasted work but not wrong. By breaking first, the code is cleaner and the exit is explicit.
  • f"Total: £{total:.2f}" — formatted output: the :.2f tells Python to show the number to 2 decimal places. £3.5 displays as £3.50 — proper money format. £ is the pound sign. Without :.2f, you might see "£3.5" which loses a mark for presentation.
  • STRUCTURE = MARKS: notice how each function does ONE thing. show_menu only displays. get_choice only reads + validates. price_of only looks up prices. The main program coordinates them. This separation (called cohesion) is what examiners reward heavily. A 50-line program with everything dumped together earns far fewer marks than this 25-line version split into focused functions.
🎯 Exam tip — the 12-mark template: When you see "write a complete program" worth 10+ marks, ALWAYS use this skeleton: (1) imports if needed, (2) 2-4 small functions each doing one job, (3) a clearly-commented "# Main program" section that calls them in a loop with a running total / list / counter, (4) a final print of the result. Even if you don't finish, marks come from the STRUCTURE — partial-credit is generous when the design is clear.
